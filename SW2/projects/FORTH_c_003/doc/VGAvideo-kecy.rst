VGA Signal Generation on ATmega2560
==================================

This project implements a software-driven VGA signal generator using the AVR
microcontroller ATmega2560. The design relies on hardware timers for precise
timing and uses interrupts only as scheduling mechanisms, not as time references.

Core Design Principle
--------------------

Interrupt latency on AVR is not deterministic due to:

- instruction completion
- interrupt masking
- /* register save/restore overhead - this is deterministic, but slow \*/ 

Therefore, **no time-critical action is started directly inside an interrupt**.

Instead:

1. Interrupt captures current timer state
2. A precise future time is scheduled using compare registers
3. The actual operation starts at a deterministic hardware event

This eliminates jitter.

Timing Architecture
-------------------

Two timers are used:

- Timer1: Horizontal timing (HSYNC, scanlines)
- Timer3: Vertical timing (VSYNC, frame)

Signals:

- HSYNC: generated by Timer1 PWM (OC1B, PB6)
- VSYNC: generated by Timer3 PWM (OC3B, PE4)

Interrupt roles:

- TIMER3_OVF: start of frame (vertical sync reference)
- TIMER3_COMPA: start of visible area
- TIMER3_COMPB: end of visible area

- /* TIMER1_OVF: start of scanline (reference point only) \*/
- TIMER1_COMPA: precise start of pixel output (visible part of line)

Visible Area Control
--------------------

The visible region is controlled by enabling/disabling rendering:

- At line 50: rendering is enabled
- At line (50 + 55*8): rendering is disabled

This is handled by Timer3 compare interrupts.

Scanline Rendering
------------------

Each scanline is split into:

1. HSYNC pulse (hardware PWM)
2. back porch (idle)
3. visible pixels (software-driven output)

The exact start of pixel output is scheduled using Timer1 COMPA interrupt.

To eliminate jitter:

- /* TIMER1_OVF does NOT render \*/
- /* it only schedules COMPA = TCNT1 + offset \*/
- TIMER1_COMPA starts with 1T jitter elimination, which should offset jitter from rjmp loop in TIMER3_COMPA interrupt

Rendering happens in TIMER1_COMPA_vect.

Critical Section Strategy
------------------------

During visible frame:

- main loop is effectively paused (TIMER3_COMPA blocks normal computing, but enables interrupts early on the start)
- rendering runs in tightly controlled interrupts, rendering interrupt jumps into TIMER3_COMPA directly (discarding iret and I flag) at end to allow compute of next line

Other interrupts:

- Keyboard interrupt is allowed
- If it delays rendering beyond threshold:
  -> scanline is skipped entirely
- if TIMER1_COMPA_vect already starte, the keyboard interrupt is posponed to the end of line (32 uS max)

This avoids horizontal jitter artifacts.

Summary
-------

- All critical timing derived from hardware timers
- Interrupts are used only to schedule future events
- Rendering starts at deterministic compare match
- Jitter is eliminated at the cost of CPU availability

Zadani:

Prima, generuju VGA signal (uz mi to chodi, ale chci to vylepsit), coz kombinuje nekolik veci:
- na spravne piny musi prichazet synchronizacni pulzy (jeden vertikalni pro celou obrazovku - PE4 - Timer 3 , druhy horizontalni pro kazdy radek - PB6 - Timer 1)  spravne delky (to ted funguje)
- pro zacatek generovani viditelnych radku je potreba na 50 scan line povolit generovani signalu (zavolat jeden interrupt)
- vygenerovat signal pro kazdou scanline az do 50 + 55*8 (pokazde ve spravnou chvili pred zacatkem radku/scanline  vyvolat treti interrupt)
- zakazat generovani signalu (zavolat druhy interrupt)

- tyto interrupty musi byt nezavisle, treti interrupt by asi generoval Timer 1, prvni a druhy Timer 3
- muj plan je, ze prvni interrupt (zacatek viditelne oblasti) ulozi registry, nacte data a povoli interrupty, ale bude ukoncen az po druhem interruptu (konec viditelne oblasti) - bude blokovat vypocet, aby treti interrupt (viditelny subradek - vodorovna linka) nemel jitter

- cele to zarizeni je graficka karta (nejakymi funkcemi navic), takze zablokovani hlavniho casu pro hlavni funkci je v poradku, veci navic muzou bezet ve zbytku casu, kdyz neni co delat, nejsou prilis dulezite a hlavne nespechaji.
- je tam jeste interrupt od klavesnice (klasicky interrupt, ktery zachova registry, ale stoji nejaky cas), ktery muze zpusobit, ze by se radek nezacal vykreslovat vcas - v takovem pripade se radek nevykresli vubec (pusobi to min rusive, nez kdyz se vykresli posunute a rozhodi dalsi radky) tento interrupt je nejdulezitejsi, ale zpusobuje ho clovek, takze prichazi zridka (rekneme max. 5x za sekundu, zatimco obrazovak se vykresli 60) - ten je osetren jinde a nebudeme se jim zabyvat

Potreboval bych pro tohle 
- vygenerovat nejaky strucny ale dostatecne podrobny popis, pro prezentaci na webu (idealne RST, nebo proste zformatovany text) 
- vygenerovat kostru programu v asembleru, ktera nastavi ty interrupty a synchronizacni pulzy. Dole pro predstavu prikladam soucasny stav, kde ty pulzy jsou spravne, interrupty nikoli (pouzival jsem pro ne jinou filozofii, ted chci tuhle novou)
- pokud by ti nebylo cokoli jasne, zeptej se, a ja doplnim dalsi detaily

				executed when the TOVn Flag, located in TIFRn, is set.
				*/
			
			; TIMSK1 = (1 << TOIE1); 
		ldi r24, (1 << TOIE1)
		sts TIMSK1, r24
		
				// ISR timer overflow
				// ISR(TIMER1_OVF_vect) {}
				// This will be called each 32 µs

	;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;

		// *****************************
		// ***** Timer3: VGA VSYNC *****
		// *****************************
		//            PE[4] D2          

			; write H, then L, read L, then H - pg.135
			; TCNT3=0;
		sts TCNT3H,r1
		sts TCNT3L,r1
		
	//	// Mode: Fast PWM, TOP=ICR3, Set on Compare, Clear on Bottom

		;  attach output pin PE4 = OC3B
			; TCCR3A = (1 << COM3B1) | (1 << COM3B0) | (1 << WGM31); 
		ldi r24, (1 << COM3B1) | (1 << COM3B0) | (1 << WGM31)
		sts TCCR3A, r24
			; TCCR3B = (1 << WGM33) | (1 << WGM32) | (1 << CS32) | (1 << CS30); // prescaler 1024
		ldi r24, (1 << WGM33) | (1 << WGM32) | (1 << CS32) | (1 << CS30)
		sts TCCR3B, r24
			; TCCR3C ignored in PWM
		
			; ICR3 = 259;   // period = 130 ticks	// // 60cols (24MHz): 389, 40cols (16MHz): 259, compare match register A (TOP) -> 16.64ms
		ldi r24, lo8(259)
		ldi r25, hi8(259)
		sts ICR3H, r25
		sts ICR3L, r24
			; OCR3B = 0;    // LOW = 1 tick = 64 µs
		sts OCR3BH, r1
		sts OCR3BL, r1

		; Enable this interrupt TIMER3_OVF_vect:
			; TIMSK3 = (1 << TOIE3);	// enable timer overflow interrupt setting vlines = 0
		ldi r24, (1 << TOIE3)
		sts TIMSK3, r24
	;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;

			; GTCCR = 0;	// clear TSM => all timers start synchronously
		out GTCCR, r1
			; UCSR1B = 0;	// brute-force the USART1 off just in case... ??
		sts UCSR1B, r1

		sei

		ret

		ret	; PS/2 will be handled separately
	1:

	.type TextVGA_TIMER3_OVF_vect_handler, @function

		sts TextVGA_scanline+1, r0
		sts TextVGA_textline, r0
		sts TextVGA_subline, r0

		lds r25, TextVGA_VRAM+1

		sts TextVGA_VRAM_ptr+1, r25

		sts TextVGA_cursor_counter, 24

Pro atmega2560 pisu v asembleru funkci, ktera je volana z preruseni a potrebuje zacit neco delat vzdy ve stejnem case, i kdyz to preruseni a jeho osetreni nabralo nejake nahodne zpozdeni

Tohle je klasický problém „deterministického startu“ z ISR na ATmega2560 – přerušení samo o sobě má nejistou latenci (aktuální instrukce, zakázaná přerušení, další ISR atd.), takže kód v něm nezačne běžet ve zcela stejném čase.

Řešení je: neřídit čas začátku podle okamžiku vstupu do ISR, ale podle hardwarového časovače.
🔧 Princip řešení

Místo:

    „přišlo přerušení → hned začnu něco dělat“

udělej:

    „přišlo přerušení → naplánuju přesný čas → čekám na něj“

✅ Varianta 1: Compare Match (nejčistší)

Použij např. Timer1:

    ISR pouze uloží referenční čas:

in   r16, TCNT1L
in   r17, TCNT1H

    Přičteš pevný offset:

adiw r16, OFFSET   ; OFFSET = počet taktů do požadovaného startu

    Zapíšeš do compare registru:

out OCR1AH, r17
out OCR1AL, r16

    Skutečná akce se provede až v compare ISR:

ISR(TIMER1_COMPA_vect):
    ; tady začneš – vždy přesně

👉 Výhoda:

    absolutně deterministické (±1 takt)

    nezávislé na zpoždění původního přerušení

✅ Varianta 2: aktivní čekání na čas (busy wait)

Pokud nechceš další ISR:

; uložený target time v r16:r17

wait_loop:
    in   r18, TCNT1L
    in   r19, TCNT1H
    cp   r18, r16
    cpc  r19, r17
    brlo wait_loop

; tady start

👉 Nevýhoda:

    blokuje CPU

    ale časově přesné

✅ Varianta 3: synchronizace na timer overflow

Použiješ overflow jako „globální časovou základnu“:

    ISR jen nastaví flag

    hlavní kód čeká na konkrétní overflow tick

⚠️ Na co si dát pozor
1. Latence ISR na AVR

Na ATmega2560:

    4 cykly: dokončení instrukce

    4 cykly: skok do ISR

        push registrů (ručně / compiler)

👉 tohle NIKDY nebude konstantní → ignoruj to a používej timer
2. Atomické čtení timeru

16bit timer:

in r16, TCNT1L
in r17, TCNT1H

(nebo použij doporučené pořadí dle datasheetu)
3. Přetečení

Musíš počítat modulo 16 bitů:

(current - target) < 0

💡 Nejlepší praxe

✔ ISR = jen:

    uložit timestamp

    naplánovat compare

    rychle pryč

✔ přesná práce = až v compare ISR
🧠 Shrnutí

Pokud chceš vždy stejný start bez jitteru, tak:

    ❌ ne: „spustím to hned v ISR“
    ✅ ano: „naplánuju přesný čas pomocí timeru“

📄 Dokumentace (RST / web)

VGA Signal Generation on ATmega2560
==================================

This project implements a software-driven VGA signal generator using the AVR
microcontroller ATmega2560. The design relies on hardware timers for precise
timing and uses interrupts only as scheduling mechanisms, not as time references.

Core Design Principle
--------------------

Interrupt latency on AVR is not deterministic due to:

- instruction completion
- interrupt masking
- register save/restore overhead

Therefore, **no time-critical action is started directly inside an interrupt**.

Instead:

1. Interrupt captures current timer state
2. A precise future time is scheduled using compare registers
3. The actual operation starts at a deterministic hardware event

This eliminates jitter.

Timing Architecture
-------------------

Two timers are used:

- Timer1: Horizontal timing (HSYNC, scanlines)
- Timer3: Vertical timing (VSYNC, frame)

Signals:

- HSYNC: generated by Timer1 PWM (OC1B, PB6)
- VSYNC: generated by Timer3 PWM (OC3B, PE4)

Interrupt roles:

- TIMER3_OVF: start of frame (vertical sync reference)
- TIMER3_COMPA: start of visible area
- TIMER3_COMPB: end of visible area

- TIMER1_OVF: start of scanline (reference point only)
- TIMER1_COMPA: precise start of pixel output (visible part of line)

Visible Area Control
--------------------

The visible region is controlled by enabling/disabling rendering:

- At line 50: rendering is enabled
- At line (50 + 55*8): rendering is disabled

This is handled by Timer3 compare interrupts.

Scanline Rendering
------------------

Each scanline is split into:

1. HSYNC pulse (hardware PWM)
2. back porch (idle)
3. visible pixels (software-driven output)

The exact start of pixel output is scheduled using Timer1 COMPA interrupt.

To eliminate jitter:

- TIMER1_OVF does NOT render
- it only schedules COMPA = TCNT1 + offset

Rendering happens in TIMER1_COMPA_vect.

Critical Section Strategy
------------------------

During visible frame:

- main loop is effectively paused
- rendering runs in tightly controlled interrupts

Other interrupts:

- Keyboard interrupt is allowed
- If it delays rendering beyond threshold:
  -> scanline is skipped entirely

This avoids horizontal jitter artifacts.

Summary
-------

- All critical timing derived from hardware timers
- Interrupts are used only to schedule future events
- Rendering starts at deterministic compare match
- Jitter is eliminated at the cost of CPU availability

👉 jen naplánuje TIMER1_COMPA

    push r17

    lds r17, TextVGA_scanline+1

    sts OCR1AL, r16

    pop r17
    pop r16
    reti

Stačí říct 👍

"dlouhý ISR přes celou oblast“ nerozbije ostatni interrupty, protoze po kratkem uvodu (ulozeni registru) interrupty povoli a tak z jejich pohledu bude jen jednodussim hlavnim programem (falesny main), ktery budou prerusovat po libosti (a ktery nebude obsahovat zadne kriticke sekce)

Zaroven odstini opravdovy hlavni program, ktery by ale stejne nic v tu dobu delat nemohl, protoze vykreslovani pokryje veskery dostupny cas zbyly po vyrizeni rezie (presunovani ukazatelu na dalsi radek a tak podobne, ono se to nezda, ale v nejhorsim pripade rezije pobere 70 taktu procesoru a naklady na ukladani registru pri volani interruptu klidne i dalsich 68+68 - ktere si takto muzu usetrit na asi 16 znaku navic) 

latenci to naopak snizi, protoze v kontrolovanem prostredi bude radkovy interrupt mit jitter jenom 1 tick a nebude muset ukladat registry (falesny main na registrech uz nezalezi) a nacitat data (protoze je falesny main na zacatku nacetl do registru a uz na ne nesaha). 

Ladeni se prizpusobi, stejne je to smycka zkusim-uvidim-upravim, zadny ladici nastroj tam bez rozhozeni casovani stejne zasahnout nemuze.

Jeste bych potreboval udelat pocatecni nastaveni tech timeru/compare a povoleni interruptu aby se to vsechno volalo spravne

Jo – takhle když to popisuješ detailněji, tak to dává smysl 👍
To už není „dlouhý ISR“ v klasickém smyslu, ale spíš:

    časově deterministický rendering loop běžící v kontextu ISR

a to je na ATmega2560 naprosto legitimní technika (reálně se tímhle stylem dělají i DVI/VGA hacky na AVR).

👉 Klíčové je, že:

    si ručně kontroluješ registry

    povolíš globální přerušení (sei) uvnitř

    a ISR se chová jako „fake main“

To je přesně ten případ, kdy to dává větší determinismus než klasický ISR model.
🔧 Co ti chybí: správné rozplánování timerů

Cíl:

    Timer1 = řádky (HSYNC + scanline timing)

    Timer3 = frame + „gating“ (visible window)

A hlavně:
👉 všechny důležité okamžiky musí být generované compare match, ne overflow
🧭 Časová osa (důležité pro nastavení)
Timer1 (HSYNC, scanline)

Každých ~32 µs:

BOTTOM → HSYNC pulse → back porch → visible → front porch → BOTTOM

Potřebuješ 3 body:

    BOTTOM (overflow) → inkrement scanline

    COMPA → začátek visible části (start renderingu)

    (volitelně) COMPB → konec visible

Timer3 (frame)

    overflow → začátek frame

    COMPA → scanline 50 (start visible)

    COMPB → konec visible

sts TCNT3L, r1

ldi r24, (1<<COM3B1)|(1<<COM3B0)|(1<<WGM31)
sts TCCR3A, r24

ldi r24, (1<<WGM33)|(1<<WGM32)|(1<<CS32)|(1<<CS30) ; /1024
sts TCCR3B, r24

ldi r24, lo8(259)
ldi r25, hi8(259)
sts ICR3H, r25
sts ICR3L, r24

sts OCR3BH, r1
sts OCR3BL, r1

sts OCR3AL, r24

sts OCR3BH, r1
